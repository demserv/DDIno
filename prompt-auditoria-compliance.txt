### 🔹 FASE 3 — MATRIZ DE RASTREABILIDADE (CORE DA AUDITORIA)

Para **CADA item** do catálogo normativo da Fase 1, gere uma linha na **MATRIZ DE RASTREABILIDADE** com a seguinte estrutura rigorosa:

| # | ID SRS | Texto literal (resumo fiel) | Onde está no código (arquivo:linha) | Evidência textual (trecho do código) | Status | Justificativa técnica |
|---|--------|-----------------------------|--------------------------------------|--------------------------------------|--------|----------------------|

**Valores permitidos no campo Status:**
- ✅ **COMPLIANT** — atendido com evidência clara e completa
- ⚠️ **PARTIAL** — parcialmente atendido (descrever exatamente o que falta)
- ❌ **NON-COMPLIANT** — não atendido (descrever a violação)
- 🚫 **MISSING** — ausência total no código
- ❓ **AMBIGUOUS** — código existe mas não há como provar conformidade objetivamente
- ⛔ **CONFLICTING** — código contradiz a SRS

**Regras inegociáveis da matriz:**
1. **Toda evidência DEVE conter arquivo + linha + trecho real**. Sem isso, o status é automaticamente `MISSING` ou `AMBIGUOUS`.
2. **Status ✅ exige prova textual direta**. Nunca por inferência.
3. **Nenhum item da SRS pode ser omitido da matriz**. Cobertura = 100%.
4. **Se você encontrar código que implementa algo NÃO previsto na SRS**, registre em seção separada chamada `CÓDIGO FORA DE ESCOPO` com mesmo nível de detalhe.

### 🔹 FASE 4 — ANÁLISE TÉCNICA ESPECIALIZADA (POR CAMADA)

Independente da SRS, audite tecnicamente o código contra **boas práticas obrigatórias para ESP32 / ESP-IDF v5.2.2 / C com guardas C++**. Para cada item abaixo, classifique como `OK / RISCO / DEFEITO / CRÍTICO` e justifique com evidência.

#### 4.1. Estrutura e Build
- Estrutura de componentes ESP-IDF
- `CMakeLists.txt` (raiz e componentes)
- `sdkconfig` / `sdkconfig.defaults`
- `partitions.csv` coerente com uso (NVS, OTA, SPIFFS/LittleFS)
- Versionamento (tags, macros de versão)
- Guardas C++ (`#ifdef __cplusplus / extern "C"`) em headers expostos a C

#### 4.2. Inicialização e Boot
- `app_main` enxuto
- Ordem de init (NVS → log → drivers → serviços → tasks → UI)
- Tratamento de `esp_err_t` em **TODA** chamada que retorna
- Logs estruturados com TAG, nível e contexto

#### 4.3. FreeRTOS / Concorrência
- Stacks adequados (sem under/over-provisioning)
- Prioridades coerentes (UI > sensores críticos > rede > log)
- Sem busy-wait, sem `vTaskDelay(0)`
- Proteção de recursos compartilhados (mutex/semaphore)
- Tratamento de bloqueios e timeouts
- Sem alocações dinâmicas em ISR

#### 4.4. Periféricos
- I2C/SPI/ADC/PWM/GPIO configurados conforme SRS
- Drivers com retry e tratamento de timeout
- Pull-up/pull-down e direções de GPIO corretos
- Frequências, modos e barramentos compatíveis

#### 4.5. Persistência
- Uso correto de NVS (namespace, keys, tipos, tamanhos)
- Tratamento de `ESP_ERR_NVS_NOT_FOUND` (default seguro)
- Wear leveling adequado, sem escritas excessivas

#### 4.6. Rede (se aplicável)
- Wi-Fi com reconexão e backoff
- TLS / certificados / endpoints conforme SRS
- Tratamento robusto de desconexão

#### 4.7. UI / Display (se aplicável)
- Biblioteca gráfica conforme SRS (LVGL, etc.)
- Telas implementadas conforme telas previstas na SRS
- Tempo de resposta de toque, threads de UI corretas
- Sem alocações pesadas dentro do loop de render

#### 4.8. Segurança / Robustez
- Watchdog habilitado e alimentado corretamente
- Tratamento de panic / brownout / reset cause
- Sanitização de entradas e limites de buffer
- Sem `strcpy`, `sprintf`, `gets` etc. sem proteção

#### 4.9. Qualidade de Código
- Nomenclatura consistente
- Sem código morto
- Sem `TODO/FIXME/XXX` pendentes em produção
- Sem números mágicos
- Sem duplicação significativa
- Documentação de funções públicas

#### 4.10. Memória
- Uso de IRAM/DRAM/PSRAM adequado
- Sem leaks evidentes
- Sem fragmentação previsível
- Stack alta water mark monitorada onde crítico
