================================================================================
0. OBJETIVO DESTE DOCUMENTO
================================================================================
Este documento transforma todos os itens não conformes, parciais, ausentes e ambíguos identificados na auditoria em um plano de ação executável para uma IA que somente codifica.

O objetivo NÃO é criar uma nova arquitetura livre.
O objetivo é corrigir o código existente para atingir compliance máximo com a SRS normativa do projeto Monitor Aquário Inteligente.

A IA codificadora deve:
1. Ler a SRS integralmente.
2. Ler o código integralmente.
3. Não inventar requisitos.
4. Não inventar arquivos se já existirem equivalentes corretos.
5. Não substituir requisito por opinião técnica.
6. Implementar apenas o que a SRS exige.
7. Auditar cada entrega antes de considerar concluída.
8. Entregar evidências objetivas por arquivo, função, teste e requisito.

================================================================================
1. REGRAS GERAIS ANTI-DELÍRIO / ANTI-ALUCINAÇÃO
================================================================================
A IA codificadora deve obedecer às regras abaixo sem exceção:

1.1. Nunca declarar um requisito como atendido sem evidência real em código.
1.2. Nunca usar expressões como "provavelmente", "deve estar", "parece", "assumo", "foi considerado" para comprovar compliance.
1.3. Sempre que implementar algo, registrar:
     - Arquivo criado/modificado.
     - Função criada/modificada.
     - Requisito SRS atendido.
     - Teste que prova atendimento.
1.4. Se um requisito depender de informação ausente na SRS ou no código, marcar como BLOQUEADO POR FALTA DE EVIDÊNCIA, não inventar.
1.5. Não alterar pinagem, hardware, baseline, interfaces, ALMs, FSMs, estados globais, nomes de requisitos ou semântica sem base textual na SRS.
1.6. Não usar Mermaid, documentos históricos, exemplos de LVGL ou comentários antigos como baseline se eles conflitarem com a SRS consolidada.
1.7. Não tratar código upstream do LVGL como código funcional do produto DDIno.
1.8. Não esconder stubs. Toda função incompleta deve ser finalizada ou explicitamente removida se fora de escopo.
1.9. Não deixar função não-void sem retorno final explícito.
1.10. Não deixar TODO, FIXME, XXX ou mock operacional em código de produção.
1.11. Não inserir delays bloqueantes em safety, UI ou drivers críticos sem justificativa normativa e sem timeout.
1.12. Não usar valores operacionais hardcoded; todo valor operacional deve vir de configuração, constante de hardware normativa ou catálogo de parâmetros.
1.13. Toda alteração deve compilar com warnings tratados como erro.
1.14. Toda entrega deve incluir uma seção AUDITORIA DA ENTREGA.

================================================================================
2. DEFINIÇÃO GLOBAL DE PRONTO — NÃO NEGOCIÁVEL
================================================================================
Uma entrega só pode ser considerada pronta se cumprir TODOS os critérios:

2.1. Compila com idf.py build sem erro.
2.2. Não há warning novo.
2.3. Não há função não-void sem return.
2.4. Não há mock operacional em produção.
2.5. Não há hardcode operacional fora de config/schema/catálogo permitido.
2.6. Não há chamada de hardware direto a partir de UI/API.
2.7. UI/API passam por command_validator.
2.8. SAFE_OFF e EMERGENCY prevalecem sobre comandos manuais.
2.9. Relés iniciam OFF.
2.10. P01 e P02 nunca acionam simultaneamente aquecedor/cooler.
2.11. SPI compartilhado usa mutex em TODAS as transações.
2.12. Falha SD não bloqueia ADC/safety.
2.13. Watchdog tem heartbeat de tasks críticas.
2.14. Transições globais registram estado anterior, novo estado, causa, ALM e timestamp.
2.15. Testes mínimos associados ao requisito foram criados ou atualizados.
2.16. RTM foi atualizada.
2.17. O relatório da entrega informa evidências objetivas.
