================================================================================

STATUS ATUAL:
NON-COMPLIANT.

PROBLEMA:
Foram observadas funções incompletas/sem retorno e trechos possivelmente corrompidos no concatenado. Em código C/C++ para ESP-IDF, função não-void sem retorno e stubs em drivers críticos tornam o firmware não confiável e impedem compliance.

AÇÕES DETALHADAS:
1. Ativar flags rígidas no CMake do projeto:
   - -Wall
   - -Wextra
   - -Werror
   - -Wreturn-type
   - -Wshadow, se viável
   - -Wswitch-enum, se viável para FSMs críticas
2. Rodar build limpo:
   - idf.py fullclean
   - idf.py build
3. Catalogar todos os erros por arquivo/função.
4. Corrigir imediatamente funções não-void sem retorno.
5. Remover stubs silenciosos em drivers críticos. Stub temporário só pode existir se:
   - retornar erro explícito ESP_ERR_NOT_SUPPORTED ou ESP_ERR_INVALID_STATE;
   - gerar log ESP_LOGE/ESP_LOGW;
   - não mascarar dado como válido;
   - estiver documentado como pendente em RTM.
6. Corrigir funções com corpo vazio que deveriam executar transação real:
   - mcp3208_read_channel()
   - pzem_read_all()
   - pzem_reset_energy()
   - acs712_init()
   - acs712_read_plug()
   - acs712_calibrate_zero()
   - ad_keypad_read()
   - ds18b20_read()
   - mcp23017_init(), se incompleta
7. Para cada driver, garantir padrão de retorno:
   - ESP_OK somente se operação real teve sucesso.
   - ESP_ERR_TIMEOUT em timeout.
   - ESP_ERR_INVALID_CRC em CRC inválido, se definido.
   - ESP_FAIL em falha genérica.
   - ESP_ERR_INVALID_ARG em argumento inválido.
8. Criar checklist firmware/docs/BUILD_EVIDENCE.md com:
   - comando executado;
   - data/hora manual do build;
   - commit/hash;
   - resultado;
   - warnings remanescentes;
   - decisão para cada warning.

ARQUIVOS IMPACTADOS:

- firmware/hal/*.c
- firmware/core/*.c
- firmware/services/*.c
- firmware/docs/BUILD_EVIDENCE.md

CRITÉRIO DE ACEITE:
- Build finaliza com sucesso.
- Nenhuma função não-void sem retorno.
- Nenhum driver crítico retorna ESP_OK sem executar operação real.

TESTES/EVIDÊNCIAS:

================================================================================

STATUS ATUAL:
NON-COMPLIANT.

PROBLEMA:
A arquitetura ainda descreve loop principal único de 50 ms. A SRS exige tasks FreeRTOS específicas com heartbeat e isolamento entre safety, sensores, controle de plugues, storage, UI, web e diagnóstico.

AÇÕES DETALHADAS:
1. Remover o loop monolítico como orquestrador principal.
2. Criar task_safety_core:
   - prioridade muito alta;
   - responsável por GlobalState, safety controller, SAFE_OFF, EMERGENCY, watchdog crítico;
   - período curto e determinístico;
   - não pode bloquear em I/O, SD, Web ou UI.
3. Criar task_sensors:
   - prioridade alta;
   - responsável por DS18B20, MCP3208, ACS712, PZEM e snapshots de sensores;
   - deve publicar dados em snapshot thread-safe;
   - falhas devem marcar health degradado/failed, não travar safety.
4. Criar task_plug_control:
   - prioridade alta;
   - responsável por relés, plug manager e comandos validados;
   - deve consumir fila de comandos já aprovados pelo command_validator;
   - deve obedecer SAFE_OFF/EMERGENCY acima de qualquer comando.
5. Criar task_storage:
   - prioridade média;
   - NVS, SD, logs, snapshots, rollback;
   - não pode bloquear safety;
   - fila assíncrona para logs.
6. Criar task_ui:
   - prioridade média;
   - LVGL tick/timer handler;
   - ViewModel real derivado do GlobalState;
   - falha da UI não pode afetar relés.
7. Criar task_web:
   - prioridade baixa/média;
   - esp_http_server;
   - endpoints /api/v1;
   - comandos enviados ao command_validator.
8. Criar task_diag:
   - prioridade baixa;
   - health matrix, métricas, heap, stack, diagnóstico.
9. Criar task_supervisor.c/h:
   - registrar heartbeat de cada task;
   - armazenar last_seen_ms;
   - classificar task crítica/não crítica;
   - acionar WDT/SAFE_OFF quando task crítica falhar.
10. Criar watchdog_guard.c/h:
   - registrar tasks no esp_task_wdt quando aplicável;
   - diferenciar falha de task_safety_core vs task_web/ui;
   - registrar reset reason.
11. app_main deve:
   - inicializar NVS/config;
   - configurar GPIO/relés em OFF;
   - iniciar barramentos;
   - iniciar GlobalState;
   - iniciar Alert Manager;
   - iniciar tasks;
   - iniciar supervisor;
   - não conter lógica operacional em while único.

ARQUIVOS IMPACTADOS:
